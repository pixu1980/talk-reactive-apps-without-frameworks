The pitch

  • Build a reactive UI stack from platform primitives, not from framework abstractions
  • Use signals and DOM parts to target precise DOM updates instead of diffing the whole tree
  • Keep view authoring expressive with tagged templates, directives and keyed repeat
  • Ship a Todo demo with persistence, filters, bulk actions and a live event log

Why now?

  • Signals are converging on a shared vocabulary for state, computed values and effects
  • DOM Parts show how native DOM ranges can behave like stable update slots
  • Tagged templates already give us static structure plus dynamic insertion points
  • Proxy + CustomEvent make state inspection and tooling straightforward

The browser is catching up

The platform is getting closer to the primitives UI frameworks have been emulating for years. The point is not "frameworks bad", it is understanding how much of the stack can move back into the browser.

Not a final standard, but a very interesting direction

  • TC39 Signals are still Stage 1, so this talk uses a home-grown runtime, not a standards API
  • DOM Parts are still a proposal, so we emulate them with comment anchors and ranges
  • Key takeaway: the platform direction is coherent enough to prototype against it today

Scope and honesty

  • This is not a framework replacement manifesto
  • This is a deliberately small runtime to study data flow, rendering and tradeoffs
  • Some parts are fine-grained, some parts are intentionally coarse and explicit
  • We will look at the actual code, the actual demo and the actual bugs
